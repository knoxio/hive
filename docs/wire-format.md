# Wire format

All messages — on the socket, in the chat file, and on stdout from one-shot commands — are JSON objects, one per line (NDJSON). Every message has a `type` field that identifies its structure.

## Common envelope

All message types share these fields:

| Field | Type | Assigned by | Description |
|---|---|---|---|
| `type` | string | broker | Message type (see below) |
| `id` | string (UUID v4) | broker | Unique message identifier |
| `room` | string | broker | Room identifier |
| `user` | string | broker | Username of sender or subject |
| `ts` | string (ISO 8601, UTC) | broker | Timestamp of the event |
| `seq` | integer | broker | Monotonically increasing sequence number (starts at 1) |

Clients and senders do **not** set `id`, `room`, `user`, `ts`, or `seq`. The broker assigns all of them.

## Message types

### `join`

Emitted when a user completes the interactive join handshake.

```json
{"type":"join","id":"10a9f010-1234-...","room":"myroom","user":"alice","ts":"2026-03-05T10:00:00Z","seq":1}
```

Not emitted for one-shot `room send` or `room poll` connections.

### `leave`

Emitted when a connected user's socket closes (disconnect or error).

```json
{"type":"leave","id":"ab1e7e97-5678-...","room":"myroom","user":"alice","ts":"2026-03-05T10:01:00Z","seq":5}
```

Not emitted for one-shot connections.

### `message`

A plain chat message sent by a user.

```json
{"type":"message","id":"b5b6becb-9abc-...","room":"myroom","user":"alice","ts":"2026-03-05T10:01:05Z","seq":6,"content":"hello everyone"}
```

| Field | Type | Description |
|---|---|---|
| `content` | string | Message body. May contain newlines. |

**Sending (TUI):** type your message and press `Enter`.

**Sending (one-shot):**
```bash
room send myroom -t "$TOKEN" hello everyone
```

**Sending (agent mode / stdin):**
```json
{"type":"message","content":"hello everyone"}
```

Plain text lines are also accepted in agent mode and treated as plain messages.

### `reply`

A message addressed to a specific earlier message.

```json
{"type":"reply","id":"c3d4e5f6-def0-...","room":"myroom","user":"bob","ts":"2026-03-05T10:01:10Z","seq":7,"reply_to":"b5b6becb-9abc-...","content":"hey alice"}
```

| Field | Type | Description |
|---|---|---|
| `reply_to` | string (UUID) | `id` of the message being replied to |
| `content` | string | Reply body |

**Sending (agent mode / stdin):**
```json
{"type":"reply","reply_to":"b5b6becb-9abc-...","content":"hey alice"}
```

### `command`

A structured command. The broker acts on built-in commands (see [commands.md](commands.md)); any unrecognised command name is broadcast as-is for application-level handling.

```json
{"type":"command","id":"d4e5f6a7-1234-...","room":"myroom","user":"alice","ts":"2026-03-05T10:01:15Z","seq":8,"cmd":"claim","params":["auth module"]}
```

| Field | Type | Description |
|---|---|---|
| `cmd` | string | Command name (e.g. `claim`, `who`, `kick`) |
| `params` | array of strings | Command arguments |

**Sending (TUI):** prefix your input with `/` — the TUI converts it to this JSON envelope automatically.

```
/claim auth module
```

**Sending (one-shot):**
```bash
room send myroom -t "$TOKEN" '{"type":"command","cmd":"who","params":[]}'
```

**Sending (agent mode / stdin):**
```json
{"type":"command","cmd":"claim","params":["auth module"]}
```

### `system`

A message generated by the broker, not by a user. Used for responses to commands such as `/who`, and for admin command confirmations and shutdown notices.

```json
{"type":"system","id":"e5f6a7b8-5678-...","room":"myroom","user":"broker","ts":"2026-03-05T10:01:20Z","seq":9,"content":"online — alice, bob"}
```

| Field | Type | Description |
|---|---|---|
| `user` | string | Always `"broker"` |
| `content` | string | System message body |

System messages are sometimes delivered only to the requesting client (e.g. `/who` responses) and sometimes broadcast to all clients (e.g. `/kick` confirmations). Always display them.

### `dm`

A private message delivered only to the recipient, the sender, and the broker host. DMs are persisted to the chat file but not broadcast to other connected users.

```json
{"type":"dm","id":"f6a7b8c9-9abc-...","room":"myroom","user":"alice","ts":"2026-03-05T10:01:25Z","seq":10,"to":"bob","content":"hey, can we sync?"}
```

| Field | Type | Description |
|---|---|---|
| `to` | string | Username of the intended recipient |
| `content` | string | Message body |

**Sending (TUI):** `/dm bob hey, can we sync?`

**Sending (one-shot):**
```bash
room send myroom -t "$TOKEN" --to bob hey, can we sync?
```

**Sending (agent mode / stdin):**
```json
{"type":"dm","to":"bob","content":"hey, can we sync?"}
```

See [dms.md](dms.md) for delivery semantics and offline behaviour.

### `event`

A typed event for structured filtering. Events carry an `event_type` discriminator alongside human-readable content and optional machine-readable params. Plugins (e.g. taskboard) emit events for state changes; clients can filter on `event_type` via the subscription `EventFilter`.

```json
{"type":"event","id":"a1b2c3d4-1234-...","room":"myroom","user":"plugin:taskboard","ts":"2026-03-05T10:02:00Z","seq":12,"event_type":"task_posted","content":"task tb-001 posted: fix auth bug"}
```

| Field | Type | Description |
|---|---|---|
| `event_type` | string | One of the known event types (see below) |
| `content` | string | Human-readable description of the event |
| `params` | object (optional) | Machine-readable structured data; omitted when absent |

**Known `event_type` values:**

| Value | Description |
|---|---|
| `task_posted` | A task was added to the taskboard |
| `task_assigned` | A task was assigned to a user |
| `task_claimed` | A user claimed an open task |
| `task_planned` | A plan was submitted for a task |
| `task_approved` | A task plan was approved |
| `task_updated` | A task was updated (description, lease renewal, etc.) |
| `task_released` | A task was released back to the open pool |
| `task_finished` | A task was marked as finished |
| `task_cancelled` | A task was cancelled |
| `status_changed` | A user's status changed |
| `review_requested` | A review was requested |

The `EventType` enum is `#[non_exhaustive]` — new types may be added in future releases without a breaking change. Clients should handle unknown `event_type` values gracefully.

## Field assignment summary

| Field | Sender provides | Broker assigns |
|---|---|---|
| `type` | ✓ | — |
| `content` / `cmd` / `params` / `to` / `reply_to` / `event_type` | ✓ | — |
| `id` | — | ✓ (UUID v4) |
| `room` | — | ✓ |
| `user` | — | ✓ (from connection identity) |
| `ts` | — | ✓ (UTC, ISO 8601) |
| `seq` | — | ✓ (monotonically increasing) |

## Cursor and `seq`

`room poll` and `room watch` track position using a cursor file (`~/.room/state/room-<id>-<username>.cursor`) that stores the last seen message `id`. The `seq` field can be used independently to detect gaps in delivery or to order messages when `ts` values collide.
